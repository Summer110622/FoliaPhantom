### Architecture and BossBar Integration
================================================================================
[002] Project Argo has successfully integrated comprehensive BossBar thread-safety support.
[003] This update ensures that all BossBar operations are redirected to the global region scheduler.
[004] The core logic resides in the new BossBarTransformer class, which utilizes ASM for bytecode rewriting.
[005] Target classes include org.bukkit.boss.BossBar and org.bukkit.boss.KeyedBossBar.
[006] Specifically, methods addPlayer, removePlayer, and removeAll are intercepted.
[007] These methods are redirected to _bb_ap, _bb_rp, and _bb_ra in FoliaPatcher.
[008] This ensures that player network updates are handled on the correct thread context in Folia.
[009] The transformation uses AdviceAdapter for safe stack manipulation.
[010] It robustly finds the plugin instance within the target class to use for scheduling.
[011] This architecture minimizes overhead by using static method redirection.
[012] The scanner has also been updated to flag classes using these BossBar APIs.
[013] This prevents unnecessary full transformation of classes that don't need it.
[014] The overall system performance is improved by skipping non-relevant classes.
[015] The deployment process now correctly places artifacts in the /argo directory.
[016] Absolute paths in pom.xml ensure consistent build outputs across different environments.

### RayTracing and World Safety
================================================================================
[102] World RayTracing operations have been made thread-safe in this major update.
[103] The new RayTraceTransformer class targets org.bukkit.World methods.
[104] Interception is implemented for rayTraceBlocks and rayTraceEntities.
[105] These operations are naturally blocking and return a RayTraceResult.
[106] To maintain compatibility, the FoliaPatcher uses a blocking helper (_b).
[107] This helper offloads the call to the appropriate region or global scheduler.
[108] It uses a CompletableFuture with a configurable timeout (API_TIMEOUT_MS).
[109] The redirected methods are _rtb and _rte for blocks and entities respectively.
[110] This allows legacy plugins to perform complex raytracing without crashing Folia.
[111] The transformer handles all parameter combinations for these methods.
[112] Stack management in ASM ensures that all 5+ arguments are correctly passed.
[113] Local variable slots are dynamically allocated to avoid conflicts.
[114] Type checking ensures that the correct method signature is matched.
[115] This update closes a significant gap in Folia compatibility for many plugins.
[116] RayTracing is essential for combat, building, and interaction logic.

### HP-PWM and Shorthand Optimizations
================================================================================
[202] We have expanded the HP-PWM (High-Performance Player & World Mirroring) system.
[203] A comprehensive suite of shorthand aliases has been added to FoliaPatcher.java.
[204] These aliases, such as _si, _ss, and _st, represent optimized entry points.
[205] By using short method names, we reduce the size of the constant pool in patched classes.
[206] This leading to smaller JAR files and faster class loading times.
[207] Performance is prioritized by bypassing human-readable long names where possible.
[208] Existing long-named 'safe' methods are now bridged by these shorthand versions.
[209] Commonly used APIs like Inventory#setItem (_si) and World#spawn (_ss) are covered.
[210] The mirroring system provides O(1) lookups for online players and worlds.
[211] Volatile caches are updated every tick via a global background task.
[212] This eliminates the need for expensive regional synchronization for read-only lookups.
[213] The shorthand naming convention is optimized for AI agents and bytecode level debugging.
[214] Human readability is intentionally compromised for maximum execution efficiency.
[215] Memory allocation is minimized by reusing existing lambda structures when possible.
[216] This is the most performant iteration of the FoliaPatcher runtime to date.

### ASM Implementation Details
================================================================================
[302] The ASM implementation uses a two-phase approach for transformation.
[303] First, ScanningClassVisitor performs a fast-scan of the bytecode constant pool.
[304] Classes that do not contain 'interesting' owners are skipped immediately.
[305] INTERESTING_OWNERS now includes BossBar and KeyedBossBar.
[306] Second, specific ClassTransformers are applied in a chain.
[307] The chain includes the new BossBarTransformer and RayTraceTransformer.
[308] Transformers use AdviceAdapter's visitMethodInsn to detect target calls.
[309] Stack manipulation involves storing arguments in locals to inject the Plugin instance.
[310] Plugin instance is found via a field scan or by checking if the class extends JavaPlugin.
[311] This ensures compatibility even when a plugin doesn't follow standard patterns.
[312] The transformation process is parallelized using Maven's multi-module build.
[313] The Shade plugin is used to bundle the core logic into CLI and Plugin jars.
[314] Signature files (META-INF/*.SF, etc.) are removed to prevent verification errors.
[315] The plugin.yml is automatically updated to include folia-supported: true.
[316] This automated branding ensures that servers recognize the patched plugin.
